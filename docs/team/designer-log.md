### 设计概述

当前6种基础武器覆盖了环绕/投射/随机/范围/锥形/光环六种攻击模式，构建空间为C(6,2)=15种组合。新增第7种基础武器"回旋镖"填补"追踪回旋双程伤害"空白 -- 投掷后自动追踪最近敌人，命中后飞回玩家手中，单次回旋造成双程伤害(飞出+飞回各一次)。回旋镖是所有基础武器中对单体DPS最高的(20@Lv3)，但投掷间隔最长(1.0s@Lv3)，定位为"Boss/精英杀手"。

新增2条进化路线：回旋镖+闪电->雷霆回旋(追踪+闪电链)、回旋镖+火焰法杖->烈焰回旋(追踪+火焰轨迹)，将进化系统从6条扩展到8条，保持每种武器恰好参与2条进化路线的完美对称。

新增2个Synergy协同：回旋镖+磁铁->磁力回旋(飞行路径吸引宝石)、回旋镖+暴击戒指->致命回旋(可暴击)，将武器+被动协同从11种扩展到13种。

### 为什么选择回旋镖

| 选择标准 | 分析 |
|---------|------|
| 攻击模式独特性 | 追踪+双程伤害 -- 6种现有武器无一具备 |
| 视觉辨识度 | V字形旋转飞行，极高辨识度 |
| Build空间扩展 | 6->7种基础武器，C(6,2)=15->C(7,2)=21种组合 |
| 进化路线扩展 | 新增2条，每种武器恰好参与2条进化路线 |
| 协同潜力 | 低频高伤与暴击戒指天然匹配，飞行路径与磁铁天然互补 |
| 竞品调研对标 | P1第5项明确提出"回旋镖=回旋追踪"作为新武器候选 |
| 实现复杂度 | 中等 -- 需要新的飞行逻辑(追踪+弧线+飞回)，但复用现有碰撞/伤害框架 |

### 决策记录

- 选择回旋镖作为Drive #18的设计工作，原因：(1) 竞品调研P1第5项明确提出"新基础武器x2（回旋镖/毒雾）"作为构建空间扩展方案；(2) 当前6种武器的攻击模式已覆盖环绕/投射/随机/范围/锥形/光环，缺少"追踪+双程"类型；(3) 回旋镖视觉辨识度极高（V字形旋转飞行），有利于像素风游戏的差异化；(4) 回旋镖的"低频高伤"定位与飞刀的"高频低伤"形成互补，丰富构建选择
- 双程伤害设计：飞出命中+飞回命中各判定一次，同一敌人可被飞出和飞回各命中一次。这创造了"回旋镖是最强单体武器"的独特定位，因为单次投掷总伤害为10(Lv3) vs 飞刀3(Lv3)
- 追踪微调机制：回旋镖不是瞬间锁定目标，而是每帧最多偏转trackAngle弧度朝目标旋转。Lv1=30度（需要大致瞄准），Lv3=60度（几乎全向追踪），模拟"从需要技巧到自动化"的成长感
- 弧线飞行：curvature参数让飞行路径呈弧线而非直线，增强"回旋"的视觉辨识度。Lv3的curvature从0.3降到0.2（更直），模拟"更熟练的投掷"
- 雷霆回旋选择闪电链而非AOE：闪电链保留了闪电的核心体验（随机电击），与追踪结合形成"追踪->命中->连锁"三段效果。每次命中40%概率触发闪电链，飞出和飞回各判定一次
- 烈焰回旋选择火焰轨迹而非范围DOT：火焰轨迹创造"移动区域控制"效果 -- 回旋镖飞过的路径成为临时危险区域。这是全新攻击维度，不与任何现有武器重叠
- 磁力回旋协同选择"飞行路径吸引宝石"而非"额外掉落"：与磁铁的"拾取范围扩大"互补 -- 磁铁扩大固定范围，磁力回旋在飞行路径创造移动拾取带
- 致命回旋协同让回旋镖可暴击：回旋镖的"低频高伤"与暴击的"概率翻倍"天然匹配 -- 每次命中暴击收益=5x2=10，是所有基础武器中单次暴击收益最高的
- 未给回旋镖设计幸运硬币协同：幸运硬币已有4个协同（待实现），再加会过载密度。未来可考虑"幸运转轮"协同
- 回旋镖的初始武器选择加入魔法师的自选池：战士(飞刀)和游侠(圣水)不变，魔法师从圣水/飞刀/闪电扩展为圣水/飞刀/闪电/回旋镖
- 本设计将构建空间从C(6,2)=15扩展到C(7,2)=21（+40%），进化路线从6条扩展到8条（+33%），武器+被动协同从11种扩展到13种（+18%）
- 回旋镖对5分钟短局的平衡影响：Lv2(约2次升级)DPS约17，在现有武器平衡范围内。无尽模式中Lv3+进化会成为后期核心武器

## 2026-04-05 — Drive #17: 幸运硬币Synergy协同扩展设计

### 状态总览

- 当前无 P0 事项，所有核心特性设计已完成并实现
- v1.3.0 已实施 Drive #16 的 3 项节奏平衡调优（EXP_TABLE加速、金币经济重构、spawner过渡阶段）
- P1 无尽模式 / P1 成就系统 / P2 幸运硬币：设计规格已完善，前端尚未实现
- **本次工作**：设计幸运硬币加入后所需的4个新Synergy协同效果，确保第7种被动实现时协同系统同步就绪

### 设计概述

当前Synergy系统覆盖6种被动(疾风靴/护甲/磁铁/暴击戒指/生命结晶/再生护符)和6种武器(飞刀/圣水/闪电/圣经/火焰法杖/冰冻光环)，共12种协同。幸运硬币(LuckyCoin)作为第7种被动加入后，需要扩展Synergy网络以覆盖新被动。本设计精选4个新协同：1个被动+被动协同(幸运硬币+暴击戒指)和3个武器+被动协同(幸运硬币+圣水/幸运硬币+火焰法杖/幸运硬币+冰冻光环)。设计原则是：每个协同必须有明确的Build定位、数值保守不破坏平衡、实现成本低（复用现有Synergy框架）。

### 为什么是4个而非更多

| 选择标准 | 分析 |
|---------|------|
| 被动+被动候选池 | 幸运硬币 + 其余6个被动 = 6种组合，选取1个最优 |
| 武器+被动候选池 | 幸运硬币 + 6种武器 = 6种组合，选取3个最优 |
| 总覆盖率 | 12(现有) + 4(新增) = 16种协同，覆盖7被动+6武器的核心组合 |
| 数值安全 | 每个新增协同效果独立，不叠加为连锁效果 |
| 实现成本 | 复用现有 `checkSynergies()` / `getWeaponBonus()` 框架，前端仅需追加CFG常量和少量逻辑 |

### 一、新被动+被动Synergy（1种）

#### crit_luckycoin: 命运赌徒

| 属性 | 数值 |
|------|------|
| 名称 | 命运赌徒 |
| 图标 | 🎲 |
| 条件 | 暴击戒指(crit) + 幸运硬币(luckycoin) |
| 描述 | 暴击时金币掉落翻倍 |
| 定位 | 暴击经济流核心协同 |

**设计理由**：
- Drive #13 的幸运硬币规格中已预告此协同："幸运硬币 + 暴击戒指 → 命运赌徒：暴击时金币掉落翻倍"
- 这是唯一一个同时连接"暴击系统"和"经济系统"的协同
- 暴击戒指提供暴击概率，幸运硬币提升暴击伤害+金币掉率，命运赌徒在此基础上让暴击时金币再翻倍
- 三者形成完整的暴击经济链：暴击率(戒指) → 暴击伤害(硬币) → 暴击金币翻倍(协同)

**CFG 常量**：
```js
crit_luckycoin: {
  name: '命运赌徒', icon: '🎲',
  req: { passives: ['crit', 'luckycoin'] },
  desc: '暴击时金币掉落翻倍',
  critGoldDouble: true
}
```

**实现要点**：
- 在金币掉落逻辑中检查：`if (hasSynergy('crit_luckycoin') && isCrit) gold *= 2`
- 金币翻倍在 luckycoin 的 goldDropBonus 之后计算（乘法叠加，非加法）
- 暴击判定已有 `isCrit` 标记，复用即可

**数值平衡**：
| 场景 | 基础金币 | +硬币3叠(+45%) | +暴击翻倍 | 总倍率 |
|------|---------|---------------|----------|-------|
| 僵尸(1宝石) | 4(3+1) | 5.8 | 暴击时11.6 | x2.9 |
| 精英骷髅(5宝石) | 8(3+5) | 11.6 | 暴击时23.2 | x2.9 |
| Boss(50宝石) | 53(3+50) | 76.85 | 暴击时153.7 | x2.9 |

**约束**：暴击率最高24%(3叠暴击戒指)，实际金币翻倍触发频率约每4-5次击杀1次。DPS提升0%（纯经济效果），不影响战斗平衡。

**为什么不选其他被动+幸运硬币组合**：
- 疾风靴+幸运硬币：移动速度与经济无天然联系，强行组合缺乏直觉
- 护甲+幸运硬币：防御与经济无关联
- 磁铁+幸运硬币：磁铁已有3个协同(magnet_crit/magnet_maxhp/lightning_magnet)，再加会过载磁铁的协同密度
- 生命结晶+幸运硬币：HP与经济无关联
- 再生护符+幸运硬币：回复与经济无关联

### 二、新武器+被动Synergy（3种）

#### holywater_luckycoin: 圣水炼金

| 属性 | 数值 |
|------|------|
| 名称 | 圣水炼金 |
| 图标 | 🧪 |
| 条件 | 圣水(holywater) + 幸运硬币(luckycoin) |
| 描述 | 圣水击杀敌人额外掉落1金币 |
| 定位 | 圣水流经济增强 |

**设计理由**：
- 圣水是环绕型武器，高频低伤，击杀数多但单次收益低
- 幸运硬币的金币掉率加成对高频低伤武器收益最低（每次击杀金币少但击杀多，总收益不变）
- "击杀额外掉落1金币"直接补偿高频击杀型武器的经济劣势
- 与 knife_crit(致命飞刀) 的"飞刀可暴击"定位类似——给特定武器一个独特的增强方向

**CFG 常量**：
```js
holywater_luckycoin: {

  req: { weapon: 'holywater', passive: 'luckycoin' },
  desc: '圣水击杀额外掉落1金币',

```

**实现要点**：
- 在圣水/雷暴圣水的击杀逻辑中：`if (hasSynergy('holywater_luckycoin')) player.gold += 1`
- 额外金币不受 luckycoin 的 goldDropBonus 影响（固定1金币，不参与乘法）
- 额外金币参与灵魂碎片转化（因为加到 player.gold 总量中）

**数值平衡**：
- 圣水击杀频率：约10-15次/分钟（中后期）
- 额外金币：10-15/分钟 = 50-75/5分钟
- 占总金币比例：约3-5%，温和但可感知

- 宝石价值+1 ≈ 经验+1/击杀 + 金币+1/击杀（通过gemToGold）
- 5分钟额外收益：约30-60宝石价值 = 30-60经验 + 30-60金币
- 温和，不破坏经验平衡

#### frostaura_luckycoin: 冰霜拾荒

| 属性 | 数值 |
|------|------|
| 名称 | 冰霜拾荒 |
| 图标 | ❄️🪙 |
| 条件 | 冰冻光环(frostaura) + 幸运硬币(luckycoin) |
| 描述 | 范围内被冰冻的敌人掉落宝石吸引范围+30px |
| 定位 | 冰冻流拾取增强 |

**设计理由**：
- 冰冻光环是控场型武器，核心是减速/冰冻而非击杀
- 冰冻的敌人通常不是立即被击杀，宝石掉落在敌人被冰冻的位置
- "冰冻敌人掉落的宝石吸引范围更大"让玩家不需要精确走到宝石位置就能拾取
- 与 frost_regen(极寒光环)的"冰冻概率+5%/s"自然联动——更多冰冻 = 更多宝石远距离吸引
- 这个协同是"品质生活"型——不增强伤害或经济产出，但让拾取更舒适

**CFG 常量**：
```js
frostaura_luckycoin: {
  name: '冰霜拾荒', icon: '❄️🪙',
  req: { weapon: 'frostaura', passive: 'luckycoin' },
  desc: '冰冻敌人宝石吸引范围+30px',
  weaponBonus: { frozenGemPullBonus: 30 }
}
```

**实现要点**：
- 在宝石(Gem)的吸引用逻辑中：如果该宝石来自被冰冻的敌人，且玩家拥有此协同，宝石的吸引距离增加30px
- 需要在Gem对象上标记 `fromFrozen: true`（在敌人死亡掉落宝石时，检查敌人是否处于冰冻状态）
- 仅影响宝石吸引距离，不影响拾取范围(pickupRange)
- 暴风雪(Blizzard)产生的冰冻也触发此协同

**数值平衡**：
- 冰冻覆盖率：约10-15%的敌人会被冰冻（Lv3冰冻光环8%/s概率，暴风雪15%/s）
- 额外吸引30px：基础拾取35px → 对冰冻敌人为65px，几乎翻倍
- 实际影响：减少5-10%的"走过去捡宝石"时间，间接提升DPS（更多时间打怪）
- 不影响经济总产出，只影响拾取效率

### 决策记录

- 选择幸运硬币Synergy扩展作为 Drive #17，原因：(1) 幸运硬币是唯一一个没有对应Synergy的已设计被动；(2) 竞品调研发现 20 Minutes Till Dawn 的 Synergy 系统是最佳Build深度方案，本项目应持续扩展Synergy网络；(3) 幸运硬币连接暴击+经济双系统，是最具协同潜力的被动；(4) 实现成本低（复用现有框架），但Build发现感提升显著
- 命运赌徒(crit+luckycoin)是 Drive #13 预告的协同，现在正式规格化。暴击翻倍金币放在 goldDropBonus 之后计算（乘法叠加），实际金币倍率=基础 x (1+goldDropBonus) x 2
- 圣水炼金选择"击杀额外1金币"而非"击杀掉落宝石"：圣水击杀频率高，额外宝石会显著影响经验平衡。固定1金币只影响经济，不影响经验
- 炼金烈焰选择"点燃敌人宝石+1"而非"点燃伤害+X"：火焰法杖已有 firestaff_armor 增强战斗，firestaff_luckycoin 增强经济形成差异化
- 冰霜拾荒选择"吸引范围+30px"而非"冰冻敌人掉更多金币"：避免与命运赌徒/圣水炼金的经济效果同质化。吸引范围是一个新颖的协同维度，与磁铁被动形成间接互补
- 暴击戒指参与4个协同（最多），是因为暴击是本项目最核心的随机机制，与多种被动/武器都有天然联系
- 未给飞刀/闪电/圣经设计幸运硬币协同：这三个武器已有完善的协同配对（各2个），再加会过载密度
- 本设计不修改任何现有数值或系统，纯新增内容，零回归风险
- 前端实现时应将幸运硬币本体实现与此4个Synergy一并完成，避免分两批导致的临时不一致

| 双武器Lv3(标准) | ~20 | 200 | ~10s | 合理 -- Boss出场后有~30秒交战窗口 |
| 双武器Lv3+进化 | ~35 | 200 | ~6s | 偏快，但进化需要两个满级武器，是回报 |
| 噩梦双Lv3 | ~20 | 400(2x) | ~20s | 合理 -- 需要更长时间但不会无限拉锯 |
| 休闲双Lv3 | ~20 | 120(0.6x) | ~6s | 快速击杀，符合休闲定位 |

**结论**：Boss HP在标准难度下200是合理的。噩梦模式400HP也能在20秒内解决。

#### 审计发现 #6：升级池在第7次后开始枯竭

| 问题 | 分析 |
|------|------|
| **选项池缩小** | 9-10次升级中，前5-6次有充足选择(12+个选项)。但第6次后：新武器可能已买完、已有武器已满级、被动可能已满叠。此时pool可能只剩3-4个选项，3选1实际上变成了2选1甚至"被迫接受" |
| **进化窗口过窄** | 进化需要两个武器都到Lv3。以最快速度：第1把武器Lv1(start) -> Lv2(up#1) -> Lv3(up#4)；第2把武器Lv1(up#2) -> Lv2(up#5) -> Lv3(up#8)。**最早第8次升级才能让两把武器满级**，此时已是~3:20。加上进化本身消耗1次升级(up#9)，**最早~3:45才能获得进化武器**，只剩~1:15享受进化 |

#### 暴击伤害倍率实现

**Player 新增字段**：
```js

  break;
```

#### 金币掉率实现

**当前金币掉落逻辑**（敌人死亡时）：
```js
// 敌人掉落金币（gem）
player.gold += gemValue;
```

**修改后**：
```js
const goldMul = 1 + (player.goldDropBonus || 0);
player.gold += Math.ceil(gemValue * goldMul);
```

**金币掉率数值分析**：

| 场景 | 基础金币 | 1叠(+15%) | 2叠(+30%) | 3叠(+45%) |
|------|---------|----------|----------|----------|
| 僵尸(1) | 1 | 2 | 2 | 2 |
| 蝙蝠(1) | 1 | 2 | 2 | 2 |
| 骷髅(3) | 3 | 4 | 4 | 5 |
| 精英骷髅(5) | 5 | 6 | 7 | 8 |
| Boss(20) | 20 | 23 | 26 | 29 |

注意：`Math.ceil()` 对低价值宝石(1)在+15%/+30%时仍为1，只有+45%时 ceil(1.45)=2。对于高价值敌人效果更显著。这是有意为之 -- 金币加成在后期击杀精英/Boss时才真正发力。

### 补充设计规格：weaponDmg 集成方案

**背景**：当前 `game.weaponDmgMul` 在 `beginGame()` 中正确赋值，但 registry.js 中各武器类的伤害计算尚未引用此乘数，导致"武器精通"商店升级实际上不生效。

**集成方案**：

所有武器在计算最终伤害时，乘以 `game.weaponDmgMul`。这是一个全局乘法器，应用于基础伤害计算之后、暴击判定之前。

```js
// 伤害计算流程（适用于所有武器）:
// baseDmg -> * weaponDmgMul -> * difficultyMul -> crit check
// 即: finalDmg = baseDmg * (game.weaponDmgMul || 1)
```

**各武器集成点**:

| 武器 | 伤害计算位置 | 集成方式 |
|------|------------|---------|
| Knife | update() 创建子弹时 bullet.dmg | `bullet.dmg *= (game.weaponDmgMul \|\| 1)` |
| HolyWater | update() 碰撞检测时 dmg | `dmg *= (game.weaponDmgMul \|\| 1)` |
| Lightning | update() 电击时 dmg | `dmg *= (game.weaponDmgMul \|\| 1)` |
| Bible | update() 碰撞检测时 dmg | `dmg *= (game.weaponDmgMul \|\| 1)` |
| FireStaff | update() 碰撞+点燃时 dmg | `接触dmg *= mul; 点燃dps *= mul` |
| FrostAura | update() 脉冲伤害时 dmg | `dmg *= (game.weaponDmgMul \|\| 1)` |
| 进化武器(Blizzard等) | 同上对应位置 | 同上 |

**为什么放在 baseDmg 之后、暴击之前**:
- weaponDmg 提升的是基础伤害质量（武器变强），应先计算
- 暴击是概率性翻倍（运气），应后计算
- 顺序：基础 -> 武器精通 -> 难度 -> 暴击，保证每层独立乘法

**数值验证**:
- 武器精通 Lv3 = x1.10，即所有武器伤害+10%
- Knife Lv3 baseDmg=3 -> 精通Lv3 -> 3.3 -> 暴击 -> 6.6
- 与10%增幅相比，5分钟短局内体感适中（不会破坏平衡）

### 新特性提案

#### 提案A：第8种敌人 -- 分裂虫 (Splitter)

**背景**：当前7种敌人覆盖了近战追踪/快速弱体/远程射击/闪烁瞬移/高耐久远程/Boss六种生态位，缺少"死亡后产生次级威胁"的机制。分裂虫填补"清怪压力倍增"生态位。

| 属性 | 数值 | 说明 |
|------|------|------|
| 尺寸 | 16x16（分裂后8x8） | 与僵尸等大，分裂后代缩小 |
| 血量 | 4 HP（分裂后1 HP） | 中等耐久，分裂后代脆弱 |
| 移速 | 50 px/s（分裂后70 px/s） | 分裂后代更快 |
| 伤害 | 1（接触） | 标准 |
| 特殊：分裂 | 死亡时分裂为2个小型分裂虫 | 核心机制 |
| 宝石价值 | 2（本体）/ 1（分裂体） | 总计4，高于普通 |
| 食物掉落 | 本体掉落🍖，分裂体不掉 | 限制回复收益 |

**CFG.ENEMY_TYPES 新增：**
```js
splitter:      { w:16, h:16, hp:4, speed:50, dmg:1, color:'#00897b', splits:true },
splitter_small:{ w:8,  h:8,  hp:1, speed:70, dmg:1, color:'#4db6ac', isChild:true }
```

**生成时机**：3分钟后（180s）混入生成池，与幽灵同期。权重：1（稀有）

**行为设计**：
1. 追踪玩家（类似僵尸但略快）
2. 死亡时在原位生成2个 splitter_small（继承面朝方向的左右偏移）
3. 分裂体不再分裂（isChild=true 防止无限繁殖）
4. 分裂体30%减速效果（视觉：体型更小，更难命中）

**设计理由**：
- 分裂虫让"清怪效率"成为关键考量 -- AOE武器（暴风雪/圣光领域）对分裂虫极有效，而单体武器（飞刀/闪电）效率较低
- 4HP本体+2x1HP分裂体=总共需要6HP的伤害，清怪压力显著提升但不碾压
- 与竞品调研中P1第6项"分裂型敌人"一致，符合版本规划

**视觉**：
- 青绿色（#00897b），与现有敌人色系区分（绿/紫/白/灰/红各有归属）
- 分裂时产生绿色液体粒子效果
- 分裂体用更浅的青色（#4db6ac）区分

## 2026-04-04 — 策划迭代28：Ban/Reroll升级选项设计

### 背景
当前升级面板固定展示3个选项，玩家无法更换不满意的选择。几乎所有主流 survivor-like（Vampire Survivors、Brotato、Holocure）都提供某种形式的重抽机制。低成本的UX改进可显著减少"被迫接受不需要的升级"的挫败感。

### 设计方案

**Ban/Reroll机制**：升级面板底部增加一个🔄按钮，点击后重新随机生成3个选项。

| 属性 | 数值 |
|------|------|
| 每次升级Reroll次数 | 1次（用完消失） |
| Reroll代价 | 免费 |
| Reroll后选项池 | 与首次相同的pool（重新shuffle） |
| 按钮位置 | 升级面板底部居中 |
| 按钮样式 | 白色文字 `🔄 换一批` |

#### CFG 常量

```js
UPGRADE_REROLL: {
  maxReroll: 1,  // 每次升级最多重抽1次
  buttonText: '🔄 换一批',
}
```

#### 实现要点

- `showUpgrade()` 中增加reroll按钮
- 点击reroll时：调用 `generateUpgrades(player)` 重新生成pool，shuffle后取3个
- 标记 `game._rerollUsed = true`，再次点击时隐藏reroll按钮
- 新一轮升级面板时重置 `game._rerollUsed = false`
- 无额外UI弹窗，点击即生效

### 决策记录
- **为什么免费而非收费**：5分钟短局，收费reroll会减少爽感，免费reroll提升体验
- **为什么1次而非无限**：防止无限reroll拖延游戏节奏
- **为什么不Ban（禁止某类选项）**：当前升级池6种武器+6种被动=12种，Ban后池太小，Reroll更实用
- **为什么不用金币消耗**：金币已有宝箱消耗途径，增加reroll金币消耗会让金币更加紧张

- 位置：进度条下方，8px monospace

- 新阶段开始时：底部弹出 `🆕 新敌人: 骷髅弓手!` 文字，2秒后消失

## 2026-04-03 — 策划迭代21：屏幕震动系统

### 背景
当前游戏击杀敌人、受伤、Boss出场等关键时刻缺乏视觉冲击力。屏幕震动（Screen Shake）是几乎所有动作游戏使用的低成本高回报反馈机制，让每次击杀"感觉"更强。配合 v0.17.0 的连击系统，连击越高震动越强，形成"越来越爽"的正反馈循环。

### 成果

**屏幕震动系统**：Camera 系统中集成震动偏移，通过 `game.shake` 参数驱动。

**CONFIG 数值**：
```javascript
SCREEN_SHAKE: {
  kill:     { intensity: 2,  duration: 0.08 },  // 普通击杀
  killBig:  { intensity: 4,  duration: 0.12 },  // 精英/Boss击杀
  hurt:     { intensity: 6,  duration: 0.15 },  // 玩家受伤
  boss:     { intensity: 8,  duration: 0.3  },  // Boss出场
  combo5:   { intensity: 3,  duration: 0.1  },  // 5连击里程碑
  combo10:  { intensity: 5,  duration: 0.15 },  // 10连击里程碑
  combo20:  { intensity: 7,  duration: 0.2  },  // 20连击里程碑
  combo50:  { intensity: 10, duration: 0.25 }   // 50连击里程碑
}
```

**震动机制**：
- `game.shake = { intensity, duration, timer }`
- 每帧在 Camera.w2s() 中叠加随机偏移：`(Math.random()-0.5)*2*intensity*(timer/duration)`
- timer 归零后自动停止（偏移归零）
- 新震动覆盖旧震动时取更大 intensity（不叠加，避免过度抖动）
- 震动不影响玩家移动和碰撞判定（纯视觉偏移）

**触发点**（6个）：
| 时机 | 震动等级 | 代码位置 |
|------|---------|---------|
| 普通敌人死亡 | kill | Enemy hp<=0 |
| 精英/Boss死亡 | killBig | Enemy hp<=0 + elite/boss判断 |
| 玩家受伤 | hurt | Player.takeDamage() |
| Boss出场 | boss | Boss spawn logic |
| 连击里程碑 | combo5/10/20/50 | 击杀连击里程碑检测 |

**Camera.w2s 修改**：
```javascript
w2s(wx,wy,canvas){
  let sx=(wx-this.x)*canvas.width/CVV + canvas.width/2;
  let sy=(wy-this.y)*canvas.height/CVV + canvas.height/2;
  // Apply screen shake
  if(game && game.shake && game.shake.timer>0){
    const factor=game.shake.timer/game.shake.duration;
    const i=game.shake.intensity*factor;
    sx+=(Math.random()-0.5)*2*i;
    sy+=(Math.random()-0.5)*2*i;
  }
  return{x:sx,y:sy};
}
```

**Game 新增字段**：
```javascript
shake: null  // { intensity, duration, timer }
```

**辅助函数**：
```javascript
function screenShake(type){
  const cfg=CFG.SCREEN_SHAKE[type];
  if(!cfg)return;
  if(!game.shake || cfg.intensity>=game.shake.intensity){
    game.shake={intensity:cfg.intensity, duration:cfg.duration, timer:cfg.duration};
  }
}
```

### 决策记录
- 震动取最大值而非叠加，避免多个同类型事件同时触发导致过度抖动
- 普通击杀震动极轻(2px/0.08s)，大量击杀时不会抖到看不清
- Boss出场震动最强(8px/0.3s)，传达"大事发生了"
- 震动是纯视觉偏移，不影响游戏逻辑坐标
- 配合连击系统：连击越高，每次击杀震动不变，但里程碑会触发额外大震动
- 震动衰减因子 `timer/duration` 让震动从强到弱自然消失

## 2026-04-03 — 策划迭代20：击杀连击系统

### 背景
当前游戏击杀敌人后仅获得固定经验宝石，无论玩家是密集战斗还是偶尔击杀，奖励完全相同。几乎所有主流 survivor-like（Vampire Survivors、Brotato、Holocure）都有连击/连续击杀奖励机制。击杀连击系统鼓励玩家主动出击而非被动躲避，是提升"爽感"最直接的单个机制。配合 v0.16.0 上线的 HUD 武器栏，连击计数器可显示在武器栏上方。

### 成果

**击杀连击机制设计**：

| 属性 | 数值 | 说明 |
|------|------|------|
| 连击窗口 | 3秒 | 3秒内无击杀则连击中断 |
| 经验加成 | combo × 5% | 每 combo +5% 经验（上限50%，即10连击封顶） |
| 金币加成 | combo ≥ 5 时 +1 金币 | 鼓励持续战斗 |
| 连击里程碑 | 5/10/20/50 | 触发特殊音效+视觉反馈 |
| 连击重置 | 超过3秒未击杀 | 连击归零 |

**CONFIG 数值**：
```javascript

  timeout: 3,        // 连击窗口（秒）
  expBonusRate: 0.05, // 每连击+5%经验
  maxBonus: 0.5,      // 最高50%加成（10连击封顶）
  goldThreshold: 5,   // 5连击开始额外+1金币
  milestones: [5, 10, 20, 50] // 里程碑连击数
}
```

**Player 新增字段**：
```javascript

this._comboTimer = 0;  // 连击计时器（倒计时）
this._bestCombo = 0;   // 本局最高连击
```

**连击逻辑**：
1. 敌人死亡 → `player._combo++`, `player._comboTimer = CFG.COMBO.timeout`
2. 每帧 → `if(_comboTimer > 0) _comboTimer -= dt`
3. 计时器归零 → `player._combo = 0`（连击中断）
4. 经验计算 → `bonus = 1 + min(combo * 0.05, 0.5)`（叠加磁铁的 expBonus）
5. 金币 → `if(combo >= 5) gold += 1`（每次击杀额外+1）

**HUD 显示**：
- 位置：武器栏上方，画面底部居中
- 仅在 `combo >= 2` 时显示（1连击不算）
- 内容：`🔥 ×N` 火焰emoji + 连击数
- 颜色：白色(2-4) → 橙色(5-9) → 红色(10-19) → 金色脉动(20+)
- 大小：20px monospace，连击越高越大（最大28px at 20+）
- 渐隐动画：连击中断后 alpha 0.5s 衰减（非立即消失）

**里程碑反馈**：
| 里程碑 | 音效 | 视觉 |
|--------|------|------|
| 5连击 | levelup音（短版） | 橙色闪屏 0.2s |
| 10连击 | boss音（短版） | 红色闪屏 0.3s + "🔥10连击!" 飘字 |
| 20连击 | victory音前2音 | 金色闪屏 0.4s + "🔥20连击!" 飘字 |
| 50连击 | victory音完整 | 金色闪屏 0.5s + "🔥50连击!!!" 大飘字 |

**结算画面新增**：
- 显示本局最高连击数：`🔥 最高连击: XX`
- 存档记录 `bestCombo`（跨局最佳）

### 决策记录
- 3秒窗口适中：太短(1s)连击太难维持，太长(5s)没有紧迫感
- 5%/combo的经验加成温和但不微不足道，10连击=50%加成是可观提升
- 金币加成从5连击开始，避免低连击就获得额外金币
- 不做伤害加成（已有暴击系统），选择经验加成让连击与升级系统联动
- HUD显示在武器栏上方（不遮挡操作区），连击低时不显示减少视觉干扰
- 里程碑音效复用现有 SFX（levelup/boss/victory），不新增音效类型
- 结算画面新增最高连击统计，给玩家另一个追求目标

## 2026-04-03 — 策划迭代16：被动道具扩展 + 移动端冲刺按钮

### 背景
当前升级池仅3种被动道具（疾风靴/护甲/磁铁），每个最多3叠。6种基础武器×3级+3被动×3叠=最多15次升级选择被动，但实际武器升级通常占多数，被动选项严重不足。玩家在多次游玩后感觉"每次升级都是一样的"，缺少构建多样性。同时移动端缺少冲刺按钮，手机玩家无法使用v0.12.0新增的冲刺闪避功能。

### 成果

**3种新被动道具**：

| 道具 | 名称 | 图标 | 效果/叠 | 最大叠层 | 定位 |
|------|------|------|---------|---------|------|
| `crit` | 暴击戒指 | 💍 | 暴击率+8% | 3叠(24%) | 进攻型 |
| `maxhp` | 生命结晶 | ❤️ | 最大HP+2 | 3叠(+6HP) | 续航型 |
| `regen` | 再生护符 | 🌿 | 每5秒回复1HP | 3叠(每5/4/3秒1HP) | 持续回复 |

**暴击戒指 (crit)**：
- 每叠 +8% 暴击率（3叠=24%）
- 暴击时伤害×2
- 适用范围：所有武器伤害（飞刀命中、闪电击中、圣水/圣经/火焰/冰冻/进化武器的接触伤害）
- 实现方式：伤害计算时 `Math.random() < critChance ? dmg*2 : dmg`
- `player.critChance = 0` 初始值，每叠 `+= 0.08`

**生命结晶 (maxhp)**：
- 每叠 最大HP+2（3叠=+6HP，总HP 14/18/20 取决于角色）
- 拾取时当前HP同步+2（立即获得满生命值增加）
- 实现方式：`player.maxHp += 2; player.hp = Math.min(player.hp+2, player.maxHp)`
- 配合护甲（减伤+1/叠），坦克流可以堆到极高生存力

**再生护符 (regen)**：
- Lv1: 每5秒回复1HP
- Lv2: 每4秒回复1HP
- Lv3: 每3秒回复1HP
- 实现方式：`player._regenTimer` 倒计时，归零时 `hp = min(hp+1, maxHp)`
- 不与食物系统冲突（食物是即时回复，再生是持续回复）

**CONFIG 数值**：
```javascript

  maxStack: 3, chancePerStack: 0.08, dmgMult: 2
},

  name:'生命结晶', icon:'❤️', desc:'最大HP+2',
  maxStack: 3, hpPerStack: 2
},

  name:'再生护符', icon:'🌿', desc:'每X秒回复1HP',
  maxStack: 3, intervals: [5, 4, 3]  // 每叠对应间隔
}
```

**升级池变更**：
- 被动道具从3种扩展到6种
- 升级面板逻辑不变（`generateUpgrades()` 自动包含新被动）
- 被动道具不在初始武器选择中出现（仅升级面板）

**暴击系统**：
- `player.critChance` 初始为0
- 伤害计算统一加暴击判断：`let finalDmg = dmg; if(Math.random() < this.critChance) finalDmg *= 2;`
- 暴击时显示红色"💥"飘字+数字（区别于普通白色飘字）
- 所有武器的伤害触发点统一加暴击判定

**移动端冲刺按钮**：
- 位置：虚拟摇杆右侧，距摇杆中心80px
- 尺寸：44×44px 圆形按钮（满足移动端最小触控尺寸）
- 外观：冰蓝色圆形底 + "⚡" 图标
- 冷却中：灰色覆盖+递减动画
- 就绪时：冰蓝色高亮脉动
- 触摸事件：`touchstart` 触发 `game.player.dash()`

**CONFIG 数值**：
```javascript

  color: '#4fc3f7',   // 冰蓝色
  cooldownColor: '#555' // 冷却中灰色
}
```

### 决策记录
- **暴击戒指**是首个进攻型被动，填补"全被动道具无攻击提升"的空白
- 24%暴击率（满叠）≈每4次攻击暴击1次，提供可观但不破坏平衡的DPS提升
- 暴击×2倍率简单直觉，不做×1.5等小数倍率
- **生命结晶**配合角色系统差异：战士(12+6=18HP)vs游侠(6+6=12HP)拉开生存差距
- **再生护符**提供持续回复，与食物(即时回复)和宝箱药水(大额回复)形成三级回复体系
- 再生间隔从5→4→3秒递减而非固定，让每叠有实质体感提升
- 3种新被动覆盖"进攻/血量/回复"三角，与现有"速度/防御/经验"互补
- 移动端冲刺按钮是v0.12.0遗漏的功能补全，冰蓝色与冲刺残影统一配色

## 2026-04-03 — 策划迭代15：冲刺闪避系统

### 背景
当前玩家唯一的防御手段是移动走位+无敌帧（受伤后1秒），缺少主动防御操作。几乎所有主流 survivor-like 游戏（Vampire Survivors、Brotato、Holocure）都有冲刺/闪避技能。冲刺闪避增加操作深度和"手残也能苟"的容错空间，是提升游戏体验感最显著的单个机制。

### 成果

**冲刺机制设计**：

| 属性 | 数值 | 说明 |
|------|------|------|
| 冲刺距离 | 80px | 约半个屏幕宽度 |
| 冲刺持续时间 | 0.15s | 极短爆发 |
| 冲刺速度 | 玩家移速×3 | 瞬间高速 |
| 冷却时间 | 2.5s | 防止滥用 |
| 无敌帧 | 冲刺期间全程无敌 | 核心价值 |
| 碰撞 | 穿过敌人（不穿过地图边界） | 增加策略性 |

**操作方式**：
- **键盘**：`Space` 键冲刺
- **移动端**：虚拟按钮（放在摇杆右侧）

**视觉反馈**：
- 冲刺时玩家精灵拉伸（水平方向×1.3）
- 3个残影（alpha递减0.4/0.2/0.1）跟随轨迹
- 冷却完毕时玩家脚下闪白光提示
- 冲刺音效：短促风声（高频→低频正弦波 0.1s）

**CONFIG 数值**：
```javascript
DASH: {
  distance: 80,       // 冲刺距离 px
  duration: 0.15,     // 冲刺持续秒
  speedMult: 3,       // 冲刺速度倍率
  cooldown: 2.5,      // 冷却秒
  afterimages: 3,     // 残影数量
  sound: { freq:[1200,300], dur:0.1, type:'sine' } // 冲刺音效
}
```

**Player 新增字段**：
```javascript
this._dashCD = 0;        // 冲刺冷却计时器
this._dashing = false;   // 是否正在冲刺
this._dashTimer = 0;     // 冲刺持续时间
this._dashDir = {x:0,y:0}; // 冲刺方向
this._afterimages = [];  // 残影数组 [{x,y,alpha}]
```

**冲刺逻辑**：
1. 玩家按下 Space → 检查 `_dashCD <= 0`
2. 记录当前移动方向为冲刺方向（无移动输入时用面朝方向）
3. 设置 `_dashing=true, _dashTimer=0.15`
4. 冲刺期间：以 speed×3 沿方向移动，无敌帧锁定，不接受玩家输入
5. 冲刺结束：`_dashing=false, _dashCD=2.5`
6. 移动端：新增冲刺按钮 UI

**HUD 显示**：
- 冲刺冷却指示器：HUD 右上角小圆形图标，冷却中灰色覆盖递减

### 决策记录
- 冲刺距离80px≈1秒移动距离，足够穿越敌人包围圈但不会瞬移半个地图
- 2.5秒冷却让玩家每波怪物攻势有约1次闪避机会，增加节奏感
- 冲刺期间无敌是核心价值，让玩家有"极限闪避"的爽感
- 不做方向冲刺（上下左右4向），而是面朝方向冲刺，操作更直觉
- 残影效果增强视觉冲击力，让冲刺"看得见"
- 移动端新增冲刺按钮而非双击，避免误触

## 2026-04-03 — 策划迭代12：音效系统设计

### 背景
游戏目前完全无声，缺乏打击反馈和氛围感。音效系统使用 Web Audio API 振荡器合成 8-bit 风格音效，零外部文件，保持单 HTML 文件架构。

### 成果

**SFX 配置表**：

| ID | 触发时机 | 音色 | 频率 | 时长 | 波形 | 说明 |
|----|---------|------|------|------|------|------|
| `hit` | 玩家受伤 | 下行短音 | 440→110Hz | 0.15s | square | 受击反馈 |
| `kill` | 敌人死亡 | 上行短音 | 200→600Hz | 0.1s | square | 击杀爽感 |
| `knife` | 飞刀发射 | 短促刺音 | 800Hz | 0.05s | sawtooth | 投掷感 |
| `lightning` | 闪电击中 | 噪声爆发 | 白噪声 | 0.12s | 白噪声 | 电击感 |
| `levelup` | 升级 | 三个上行音阶 | C5-E5-G5 | 0.3s | square | 成就感 |
| `pickup` | 拾取宝石/食物 | 短促叮声 | 880Hz | 0.08s | sine | 收集感 |
| `chest` | 开箱 | 上行琶音 | 440→660→880Hz | 0.25s | triangle | 惊喜感 |
| `boss` | Boss出场 | 低沉警告 | 110Hz×3 | 0.6s | sawtooth | 危机感 |
| `freeze` | 冰冻触发 | 高频碎裂 | 1200→400Hz | 0.15s | sine | 冰碎感 |
| `gameover` | 死亡/失败 | 下行长音 | 440→110Hz | 0.8s | sawtooth | 失败感 |
| `victory` | 通关 | 上行凯旋 | C5→E5→G5→C6 | 0.5s | square | 胜利感 |

**实现方案**：
- `SFX` 全局对象，`init()` 在首次用户交互时创建 `AudioContext`（满足浏览器自动播放策略）
- `play(id)` 方法：根据 ID 查表创建 `OscillatorNode` + `GainNode`，播放后自动断开
- 音量控制：`masterVolume = 0.3`（默认30%音量，避免太吵）
- 白噪声实现：`AudioBuffer` 填充随机采样值

**CONFIG 数值**：
```javascript
SFX: {
  enabled: true,
  volume: 0.3,
  sounds: {
    hit:      { freq:[440,110], dur:0.15, type:'square' },
    kill:     { freq:[200,600], dur:0.10, type:'square' },
    knife:    { freq:800,       dur:0.05, type:'sawtooth' },
    lightning:{ noise:true,     dur:0.12 },
    levelup:  { freq:[523,659,784], dur:0.1, type:'square', seq:true },
    pickup:   { freq:880,       dur:0.08, type:'sine' },
    chest:    { freq:[440,660,880], dur:0.08, type:'triangle', seq:true },
    boss:     { freq:110,       dur:0.6,  type:'sawtooth', repeat:3 },
    freeze:   { freq:[1200,400],dur:0.15, type:'sine' },
    gameover: { freq:[440,110], dur:0.8,  type:'sawtooth' },
    victory:  { freq:[523,659,784,1047], dur:0.12, type:'square', seq:true }
  }
}
```

**触发点**（10个）：
1. `player.takeDamage()` → `SFX.play('hit')`
2. 敌人 `hp<=0` → `SFX.play('kill')`
3. `Knife.update()` 发射时 → `SFX.play('knife')`
4. `Lightning.update()` 电击时 → `SFX.play('lightning')`
5. `player.addExp()` 返回 true → `SFX.play('levelup')`
6. 宝石被收集 → `SFX.play('pickup')`
7. 宝箱开箱 → `SFX.play('chest')`
8. Boss 生成 → `SFX.play('boss')`
9. 冰冻触发 → `SFX.play('freeze')`
10. `endGame()` → `SFX.play('gameover')` 或 `SFX.play('victory')`

### 决策记录
- 8-bit 合成音与像素风视觉风格统一
- 不使用外部音频文件，保持单文件零依赖架构
- `AudioContext` 延迟创建（首次交互时），满足 Chrome 自动播放策略
- 音效数量控制在10个，覆盖核心反馈但不过度
- 飞刀和闪电有专属音效，其他武器（圣水/圣经/火焰/冰冻）使用通用 kill/pickup 音效
- 未来可扩展：BGM 循环、更多武器专属音效

| 属性 | 数值 |
|------|------|

**CONFIG 数值**：
```javascript
ghost: { w:12, h:12, hp:2, speed:55, dmg:1, color:'#b0bec5', phaseShift:true, teleport:true }
```

**生成时机**：
- 2分钟（120s）后开始混入生成池
- 出现在 `getSpawnRate()` 的 180s+ 阶段，与僵尸/蝙蝠/骷髅混编
- 生成权重低于僵尸，高于骷髅（不常出现，但每次出现都有威胁感）

**行为设计**：
1. **追踪**：像僵尸一样追踪玩家，但速度更快
2. **闪烁（Phase Shift）**：每2秒进入1秒"半透明"状态
   - 视觉：alpha在0.2-0.5间脉动，给玩家"快打我/等一下"的节奏提示
   - 机制：半透明期间受到的伤害×0.5（向下取整，1伤害→0伤害）
   - 战术：玩家需要看准实体状态时攻击
3. **瞬移（Teleport）**：HP降至1时触发一次
   - 闪现到玩家背后80-120px处（反方向随机偏移±30°）
   - 瞬移后有0.5秒实体化延迟（不造成伤害），给玩家反应时间
   - 每只幽灵只触发一次瞬移

**食物掉落**：
- 幽灵掉落：🍞 面包（白色 `#e0e0e0`），与现有食物系统共用拾取逻辑

### 决策记录
- 幽灵填补"不可预测"生态位——现有4种敌人行为都是确定性追踪，幽灵通过闪烁+瞬移打破节奏
- HP=2配合闪烁减伤，实际需要2-3次命中才能击杀（闪烁期间打不疼），延长了交战时间
- 瞬移只在HP=1时触发一次，避免反复瞬移让玩家烦躁
- 生成在2分钟后混入，此时玩家已有1-2种武器，有能力应对闪烁机制
- 尺寸12×12比蝙蝠略大，保证在移动端也能看清闪烁状态

## 2026-04-03 — 策划迭代14：localStorage 存档系统

### 背景
当前游戏无持久化，每次刷新后所有数据丢失。localStorage 存档让玩家看到跨局进步感（最高分、总击杀、解锁角色），增加重玩动力。

### 成果

**存档数据结构**：
```javascript
{
  version: 1,               // 存档版本号，未来扩展用
  bestScore: 0,             // 最高分（击杀数）
  bestTime: 0,              // 最长存活时间（秒）
  totalKills: 0,            // 总击杀数（跨局累积）
  gamesPlayed: 0,           // 游玩次数
  characters: {             // 每角色最佳记录
    mage:    { bestScore:0, bestTime:0 },
    warrior: { bestScore:0, bestTime:0 },
    ranger:  { bestScore:0, bestTime:0 }
  }
}
```

**存取时机**：
| 时机 | 操作 |
|------|------|
| 游戏开始 | 读取 `roguelike_survivor_save`，解析为对象 |
| 结算画面 | 更新 bestScore/bestTime/totalKills，写回 localStorage |
| 标题画面 | 显示"最高击杀: XX"和"最长存活: X:XX" |

**CONFIG**：
```javascript
SAVE: {
  key: 'roguelike_survivor_save',
  version: 1
}
```

**结算画面变更**：
- 新增"🏆 最高记录"区域显示历史最佳
- 本次成绩超过记录时高亮显示（金色闪烁）
- 新增"本次/最佳"对比

**标题画面变更**：
- 在"5分钟，活下来"下方新增一行统计文字
- 显示：最高击杀 | 最长存活 | 游玩次数
- 无存档时不显示（首次游玩）

### 决策记录
- 只存统计数据，不存游戏进度（5分钟短局不需要断点续玩）
- 每角色独立记录（鼓励尝试不同角色）
- totalKills 累积值给玩家长期目标感
- 使用 JSON 序列化，简单可靠
- 不存武器/升级信息，保持存档结构简单
- 存档 key 包含游戏名前缀，避免与其他项目冲突
