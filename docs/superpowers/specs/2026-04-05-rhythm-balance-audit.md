# 节奏与经济平衡审计 (Rhythm & Economy Balance Audit)

**版本**: v1.2.4 | **日期**: 2026-04-05 | **Drive**: #16

---

## 1. 审计范围

本审计覆盖标准5分钟(Normal难度)游戏体验的以下维度：

- 经验获取与升级节奏（EXP_TABLE vs 击杀产出 vs 时间）
- 金币经济（局内获取 vs 消耗 vs 灵魂碎片转化）
- 击杀节奏（敌人生成密度 vs 瀑布曲线）
- Build完成度（升级频率 vs 进化窗口 vs 池枯竭）

审计方法：基于 `config.js` / `spawner.js` / `game.js` / `registry.js` 的实际代码，进行数学模拟计算。

## 3. 关键审计发现

### 发现 #1: 金币经济严重通胀

**当前状态**:
- 每次击杀获得 10 金币 (game.js 第444行 `player.gold += 10 + comboGold`)
- 5分钟 ~336 击杀 = ~3360 金币
- 5连击以上每击杀额外 +1 金币
- 高 combo 时预估: ~3500 金币/局

**消耗分析**:
- 宝箱: 20金币/个, 90s刷新, 最多2个 = 最多40金币消耗
- 金币消耗率: 40/3500 = **仅 1.1%**

**问题**: 金币几乎无限积累，宝箱20金币的价格毫无压力。玩家永远不会为"是否开箱"做决策。

**灵魂碎片影响**:
- 3500 x 30% = 1050 灵魂碎片/局（无商店加成）
- 810 / 1050 = 1.3 局即可全满商店（贪婪之心3叠）
- **1.3局全满所有商店升级**，缺少长期追求

### 发现 #2: 灵魂碎片获取过快

**全满商店成本**: 20+40+80 + 20+40+80 + 15+30+60 + 25+50+100 + 30+60+120 + 15+30+60 = 810 灵魂碎片
**每局平均获取**: ~1050 精灵碎片
**全满所需局数**: ~0.77局

即：如果玩家每局正常打完5分钟，**不到1局就能全满所有商店升级**。这大大削弱了商店系统的"跨局成长"感。

### 发现 #3: 经验曲线前松后紧

**前2分钟**:
- 击杀 ~70, 经验 ~90 (不含加成)
- 达成 Lv4-Lv5（需要 77 经验，约在 1:10-1:15）
- 仅 3-4 次升级，选择空间极小

**2:00-3:00**:
- 击杀 ~60, 经验 ~210
- 达成 Lv7-Lv8（需要 235 经验）
- 这个1分钟获得 2-3 次升级

**3:00-5:00**:
- 击杀 ~206, 经验 ~350
- 达成 Lv10-Lv11

**问题**: 前2分钟只有3-4次升级，玩家还在熟悉操作、构建尚未成型。第3分钟突然加速，但敌人也同时变强。缺少"先用Build压制变强的敌人"的正反馈循环。

### 发现 #4: 2:00-3:00 生成密度跳跃

**1:00-2:00**: interval=1.5s, count=2, 生成速度=1.33/s
**2:00-3:00**: interval=1.0s, count=3, 生成速度=3.0/s (+125%)

问题：2分钟是敌人类型质变的时间点（骷髅+幽灵加入），同时生成速度翻倍。玩家从"轻松"突然进入"压力大"模式，缺少过渡。

### 发现 #5: 升级池后期枯竭

**选项池分析**:
- 新武器: 最多5个可选（初始已占1个）
- 武器升级: 最多6个（每武器3级）
- 被动: 最多6个（每被动3叠）
- 进化: 最多6条
- 总选项池: ~17个理论最大

**实际情况**: 10-11次升级中，到第8-9次时池已明显缩小（新武器买完、部分满级、部分被动满叠）。到第10次升级可能只有3-4个选项可选。配合Reroll也只有1次，无法解决"被迫接受"问题。

### 发现 #6: 进化窗口过窄

进化条件：两个满级武器同时持有。

最快进化时间线：
- 初始武器 Lv1 -> Lv2(up#1) -> Lv3(up#4): 第4次升级时满级
- 第2把武器 Lv1(up#2) -> Lv2(up#5) -> Lv3(up#8): 第8次升级时满级
- 进化(up#9): 第9次升级
- 第9次升级预估时间: ~3:20

问题：进化获得时间约3:20-3:45，Boss在4:30出现。玩家只有约45-75秒享受进化武器，而进化是核心Build完成的高潮。

## 4. 建议调优方案

### 调优 A: EXP_TABLE 前期加速

**目标**: 前2分钟获得5-6次升级（而非3-4次），让开局更快进入构建节奏。

```js
// 当前
EXP_TABLE: [0, 10, 15, 22, 30, 40, 52, 66, 82, 100, 120, 144, 172, 206, 246]

// 建议调优
EXP_TABLE: [0, 8, 12, 18, 24, 32, 42, 55, 70, 88, 108, 132, 160, 195, 240]
```

**效果对比**:

| 等级 | 当前阈值 | 调优阈值 | 差异 | 效果 |
|------|---------|---------|------|------|
| Lv2 | 10 | 8 | -2 | 更快第1次升级 |
| Lv3 | 15 | 12 | -3 | 更快第2次升级 |
| Lv4 | 22 | 18 | -4 | 前45秒内可能已到Lv4 |
| Lv5 | 30 | 24 | -6 | 1分钟内可能到Lv5 |
| Lv6 | 40 | 32 | -8 | 前2分钟内到Lv6 |
| Lv7 | 52 | 42 | -10 | |
| Lv8 | 66 | 55 | -11 | |
| Lv9 | 82 | 70 | -12 | |
| Lv10 | 100 | 88 | -12 | |
| Lv11-15 | 不变或微调 | - | 保持后期升级节奏 |

**预期效果**:
- 前2分钟升级次数: 3-4 -> 5-6
- 5分钟总升级次数: 9-10 -> 11-12
- 进化窗口: 3:20-3:45 -> 2:45-3:15（提前约30-60秒）

### 调优 B: 金币经济重构

**目标**: 金币成为有意义的资源，而非无限膨胀的数字。

**方案 B1（推荐）: 基础金币 3 + 宝石价值加入**

```js
// CFG 新增
GOLD: {
  perKill: 3,             // 每次击杀基础金币（当前硬编码10，改为3）
  gemToGold: true,       // 宝石价值也加到金币
1:1比例
}

// game.js 修改
// 当前: player.gold += 10 + comboGold;
// 改为: player.gold += 3 + comboGold + gemValue;
```

**效果分析**:
- 5分钟基础金币: 336 x 3 = 1008（而非3360）
- 宝石金币: ~650 x 1 = ~650
1:1
- 总金币: ~1658（含combo加成）
- 宝箱消耗: 20/个 x ~2 = 40 (不变)
- 金币利用率: 40/1658 = 2.4%（比当前1.1%好，但仍很低）
- 灵魂碎片: 1658 x 30% = 497（比当前1050低）
—— 降低了53%
- 全满商店所需局数: 810/497 = 1.63局（比当前0.77局多）

**为什么3而非更低**: 3是一个"看起来有意义"的数字。低于3会让玩家感觉"打了半天没金币"。配合宝石额外收入，总金币仍可达~1600/局。

**方案 B2（保守）: 仅将基础金币从10改为5

```js
GOLD: { perKill: 5, gemToGold: false }
```

- 总金币: ~1680（含combo和宝石）
- 篮片利用率: ~2.4%
- 灵魂碎片: 504
- 全满局数: 1.6局

**为什么推荐B1而非B2**: B1同时解决两个问题（金币通胀 + 灵魂碎片过快），而且让"宝石"成为有意义的资源（不仅仅是经验来源）。B2只解决金币问题。

### 调优 C: spawner.js 过渡阶段

**目标**: 平滑 2:00-3:00 的生成密度跳跃。

**当前 spawner.js**:
```js
if (elapsed < 60)  return { interval: 2.0, count: 2, types: ['zombie'] };
if (elapsed < 120) return { interval: 1.5, count: 2, types: ['zombie', 'bat'] };
if (elapsed < 180) return { interval: 1.0, count: 3, types: ['zombie', 'bat', 'skeleton', 'ghost'] };
// 120->180: count 2->3 (+50%)
```

**建议新增 150-180 过渡阶段**:
```js
if (elapsed < 60)  return { interval: 2.0, count: 2, types: ['zombie'] };
if (elapsed < 120) return { interval: 1.5, count: 2, types: ['zombie', 'bat'] };
if (elapsed < 150) return { interval: 1.2, count: 2, types: ['zombie', 'bat', 'skeleton'] };
// 新增: 150-180过渡期，仅加入skeleton，不加ghost
if (elapsed < 180) return { interval: 1.0, count: 3, types: ['zombie', 'bat', 'skeleton', 'ghost'] };
// 180->150-180: count 2->3 的跳跃被分散到两步
  120->150(count=2) -> 150->180(count=2->3)
```

**效果**: 2分钟新增骷髅但保持count=2，3分钟新增幽灵时count升到3。密度变化更平缓。
